## Ghi chú đặc thù module
- **Read-only module:** KHÔNG có nút Tạo/Sửa/Xóa ở bất kỳ đâu. Nếu thấy CUD → FAIL (scope sai).
- **Auto-refresh 60s (FR-I-CROSS-02):** Timer chạy ngầm. Khi smoke, **không chờ 60s**, chỉ verify sau click `Làm mới` lần đầu các KPI reload (spinner xoay rồi dừng).
- **Landing page sau login:** Dashboard thường là trang đích mặc định của role CB_NV/CB_PD. Nếu landing là `/403` → verify role account (role không có permission dashboard = PASS riêng, không nhầm với auth fail).
- **KPI = 0 toàn bộ** KHÔNG = FAIL smoke. Nếu DB chưa có data nguồn (chạy smoke Lệnh 1 round mới) → KPI = 0 là expected, chỉ note để Lệnh 3 tạo data. Chỉ FAIL khi thẻ KPI render lỗi (spinner kẹt / toast `Lỗi tải dữ liệu` xuất hiện).
- **Drill-down không test ở smoke** — để functional test (DASH-016..019).

## Bước 2: VÀO MODULE

### 2a. Verify menu Dashboard trong sidebar (hoặc đã ở landing Dashboard)

1. `$B snapshot -i` sidebar
2. Grep label `Tổng quan` / `Dashboard` / `Trang chủ` — phải xuất hiện ít nhất 1 trong các label này
3. Nếu landing đã là Dashboard → skip click, verify trực tiếp URL + elements ở 2b
4. Nếu landing khác → click label Dashboard tìm thấy ở snapshot

| Kết quả | Action |
|---------|--------|
| ✅ Menu Dashboard visible (hoặc landing mặc định là Dashboard) | → 2b |
| ❌ Không thấy label Dashboard nào trong sidebar + landing `/403` | Verify role (CB_NV phải có). Role đúng mà thiếu → **FAIL**, báo FE |

*(Module này không có submenu — bỏ qua check submenu.)*

### 2b. Verify Dashboard render đủ element

1. Đợi page render (< 5s — FR-I-CROSS-02 có thể gọi nhiều API song song cho 7 KPI + 2 chart)
2. Verify URL khớp route expect (ghi nhận route thật vào Metadata sau lần đầu)
3. **Snapshot + grep các element bắt buộc:**
   - `$B snapshot` → grep `"Tổng quan hệ thống"` / `"Cập nhật lúc"` / `"Làm mới"`
   - Grep các nhãn KPI: `"hỏi đáp"`, `"vụ việc"`, `"đang xử lý"`, `"hoàn thành"`, `"đào tạo"`, `"chuyên gia"` / `"tư vấn viên"` — đủ 7 thẻ
   - Grep `"Áp dụng"`, `"Xóa bộ lọc"` (filter bar)
   - Grep 2 KPI bổ sung: `"bổ sung"` hoặc `"thời gian xử lý"`
4. Verify ≥2 `<canvas>` / `<svg>` cho 2 biểu đồ (UC8 + UC9)
5. Chụp `dashboard-page.png`

| Kết quả | Action |
|---------|--------|
| ✅ URL đúng + đủ 7 thẻ KPI + 2 KPI bổ sung + 2 biểu đồ + filter bar | → Bước 3 |
| ❌ Redirect `/login` | Session lỗi, báo dev |
| ❌ Page trắng >10s | **FAIL**, báo BE (API KPI timeout) |
| ❌ Thiếu thẻ KPI (<7) / thiếu 1 biểu đồ / thiếu filter bar | **FAIL**, note chi tiết + screenshot |
| ❌ Spinner xoay vĩnh viễn trên 1 thẻ KPI >15s | **FAIL** — API KPI đó không trả, báo BE (note endpoint qua `$B network`) |

## Bước 3: KIỂM TRA LỖI NGẦM

1. `$B console --errors`
2. `$B network` → grep 4xx/5xx **cho các endpoint KPI** (thường `/api/v1/dashboard/*` hoặc `/api/v1/kpi/*`)
3. `$B snapshot` → grep: `"Lỗi tải dữ liệu"`, `"Validation failed"`, `"Không thành công"`, `"Có lỗi xảy ra"`, `"undefined"`, `"NaN"`

| Kết quả | Verdict |
|---------|---------|
| Sạch (hoặc KPI = 0 + text "Chưa có dữ liệu" — INFO hợp lệ) | ✅ PASS |
| Warning deprecation / 401 single (auth timing) | ⚠️ WARN |
| `TypeError`/`undefined` trong value / ≥1 lỗi 5xx ở endpoint KPI / toast đỏ "Lỗi tải dữ liệu" | ❌ FAIL |
| KPI value hiển thị `NaN` / `null` / `undefined` (không phải `0`) | ❌ FAIL — format bug |

## Crash / Retry Rule

Browse crash (`about:blank` / `Target page has been closed` / `timed out`) — xem [CLAUDE.md Rule 7](../../CLAUDE.md):

- **Lần 1:** phân loại theo [CLAUDE.md Rule 9](../../CLAUDE.md) → nếu là REAL CRASH → cleanup + retry 1 lần
- **Lần 2:** **STOP ngay**, mark BLOCKED, báo user. Không retry lần 3.
